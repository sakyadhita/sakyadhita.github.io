# Plan: Investigate Replacing InteractiveMap with Native Astro Components

## Objective

Determine whether it is feasible and beneficial to replace `InteractiveMap.tsx` and its React dependencies (`react-simple-maps`, `@radix-ui/react-tooltip`) with a native Astro solution (e.g., vanilla SVG, Leaflet.js, or pure CSS/HTML).

## Current Implementation Analysis

`InteractiveMap.tsx` currently provides:

1. **SVG World Map Rendering:** Uses `react-simple-maps` to fetch a TopoJSON file (`https://cdn.jsdelivr.net/npm/world-atlas@2/countries-110m.json`) and render it as an SVG.
2. **Pan & Zoom:** Uses `<ZoomableGroup>` to allow users to interact with the map.
3. **Dynamic Markers:** Maps over an array of `markers` (passed from `HomeBody.astro`, sourced from the `branch` content collection) and plots SVG paths based on latitude/longitude.
4. **Tooltips on Hover/Click:** Wraps markers in a Radix UI Tooltip (`@radix-ui/react-tooltip` via `src/components/ui/tooltip.tsx`). The tooltip displays HTML content generated by `marked.parse()` in `src/pages/index.astro`.

## Challenges with Native Astro Replacement

Replacing this entirely with native Astro components presents several significant challenges:

1. **Client-Side Interactivity (Pan/Zoom):** Astro components generate static HTML. To provide panning and zooming of a world map, substantial client-side JavaScript is required. `react-simple-maps` handles this via D3-geo and React state. Rewriting this in vanilla JS (or a Web Component) would involve porting complex D3 projection and zoom math.
2. **Dynamic TopoJSON Rendering:** Fetching and parsing the TopoJSON file to generate the SVG paths (`<path d="...">`) on the client side requires a library like D3.js or topojson-client.
3. **Coordinate Projection:** Converting geographic latitude/longitude coordinates from the markdown frontmatter into exact X/Y coordinates on the SVG map requires projection math (e.g., Mercator or equivalent). `react-simple-maps` handles this automatically. Doing this in vanilla JS requires importing D3-geo.
4. **Accessible Tooltips:** Radix UI provides highly accessible, robust tooltips that handle positioning, collision detection, and keyboard navigation. Replacing this requires pulling in another library (like Floating UI) or writing complex vanilla JS.

## Alternatives Considered

1. **Static SVG with Vanilla JS Tooltips:**
   - _Pros:_ Zero framework JS. Fast load.
   - _Cons:_ Difficult to implement pan/zoom. Projecting lat/long onto a static SVG is hard without runtime math. If the map resizes responsively, the projections must update.
2. **Leaflet.js (Vanilla):**
   - _Pros:_ Robust mapping solution, handles pan/zoom and markers perfectly. Non-React.
   - _Cons:_ Heavier payload than the current SVG approach. Requires tile servers (which changes the visual style from a clean, abstract vector map to a more traditional street/terrain map). Less styling control over the landmasses compared to `react-simple-maps`.
3. **Mapbox GL JS / MapLibre:**
   - _Pros:_ Highly customizable vector maps.
   - _Cons:_ Overkill for a simple point-display map. Often requires API keys and has a steep learning curve.

## Conclusion & Recommendation

While minimizing React usage is a core goal of the project, **replacing `InteractiveMap.tsx` with a native Astro component is currently NOT recommended.**

The effort required to rebuild the coordinate projection, SVG rendering from TopoJSON, panning/zooming math, and accessible tooltips in vanilla JavaScript outweighs the benefits. The current React implementation is well-encapsulated and leverages specialized libraries that do this job efficiently.

Instead of replacing the component, the focus should be on **fixing the current TypeScript errors** associated with it:

- The missing type definitions for `react-simple-maps` can be resolved by installing `@types/react-simple-maps`.
- The `side` prop error on `TooltipContent` can be fixed by ensuring the `src/components/ui/tooltip.tsx` component correctly defines and passes down the `side` property if it's wrapping a Radix primitive that accepts it.
- The `dangerouslySetInnerHTML` usage is correct and necessary given the HTML output from `marked`.

**Action Plan:**

1. Leave `InteractiveMap.tsx` as a React component (`client:visible` in `HomeBody.astro` is appropriate).
2. Fix the TypeScript errors by adjusting the component props and installing missing type definitions during the next regular dependency update cycle.
